## Problem Statement

Stage 0 Gate Criterion #2 requires data contracts per layer. The Gold layer has no implementation yet, but the contract must define what features will be created, how they're aggregated, and what leakage controls exist before implementation begins.

---

## Context

Gold layer transforms Silver (normalized clinical data) into model-ready feature sets. Per IMPLEMENTATION_GUIDE.md, Gold must provide:
- Feature definitions (explicit, documented)
- Time windows (configurable, leakage-safe)
- Aggregations (clinically meaningful)
- Schema freeze (no changes during training)
- Lineage (full provenance)

This is the most critical contract for ML validity - leakage here invalidates all downstream work.

---

## Acceptance Criteria

- [ ] Contract document created (`contracts/gold_contract.md` or similar)
- [ ] Specifies input tables (from Silver)
- [ ] Specifies output feature table schema
- [ ] Lists all features with definitions (name, type, description, derivation)
- [ ] Documents time window logic (e.g., "first 2 hours after triage")
- [ ] Documents aggregation methods (mean, min, max, count, etc.)
- [ ] Explicitly documents leakage controls with rationale
- [ ] Documents target variable (admission outcome) derivation
- [ ] Documents train/val/test split strategy (temporal, random, etc.)

---

## Approach

1. Define prediction task precisely (predict admission at time T using data up to time T)
2. Define feature window (e.g., first 2 hours of ED vital signs)
3. Define aggregations per vital sign
4. Document leakage risks and controls
5. Create formal contract document

---

## Open Questions

- [ ] What prediction window? (at triage? after 2 hours? at disposition?)
- [ ] What aggregations? (min/max/mean/std/count/last?)
- [ ] Train/test split: temporal (year-based) or random?
- [ ] Should features include demographics or vital signs only?
