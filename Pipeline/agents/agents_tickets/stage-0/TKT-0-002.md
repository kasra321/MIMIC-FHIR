## Problem Statement

Stage 0 Gate Criterion #2 requires data contracts per layer. The Bronze layer currently has an ad-hoc implementation (`create_db.py`) that loads 10 tables, but no formal contract specifying:
- What tables must exist
- What columns are required
- What invariants must hold (immutability, idempotency)
- What the schema versioning strategy is

---

## Context

Bronze layer ingestion exists (`create_db.py` loads 10 FHIR resource types into DuckDB), but this was exploratory work done before contracts were defined. This ticket formalizes what the Bronze layer must provide.

**Current state:**
- 10 tables loaded (organization, location, patient, encounter, condition, procedure, observation, observation_vital_signs, medication_dispense, medication_statement)
- ~22M total records across tables
- No schema versioning
- No immutability guarantees documented

---

## Acceptance Criteria

- [ ] Contract document created (`contracts/bronze_contract.md` or similar)
- [ ] Lists all required tables with purpose
- [ ] Specifies required columns per table (at minimum: id, resourceType)
- [ ] Documents immutability requirement
- [ ] Documents idempotency requirement
- [ ] Documents schema versioning strategy (even if "v1.0 - not yet versioned")
- [ ] Documents row count validation approach
- [ ] Existing `create_db.py` assessed against contract (gaps identified)

---

## Approach

1. Review the current 10 tables in `mimic_fhir.duckdb`
2. Document what currently exists as the "de facto" contract
3. Identify gaps between current implementation and Bronze layer requirements from IMPLEMENTATION_GUIDE.md
4. Create formal contract document
